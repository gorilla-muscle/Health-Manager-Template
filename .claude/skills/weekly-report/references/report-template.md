## md の記載規則

- Mermaid ブロック必須(文章のみ不可)。レンダリング保険として同データの表を併記
- 増減率の計算ソースは `data/exercise-log/` のみ。定義は programming.md §5、1RM は Epley 式のみ
- 評価(§4〜6)は数値根拠なしの記載を禁止
- md 側は skill セクション番号・DB 表名・出典ラベルの記載 OK(トレーサビリティ確保)
- md の冒頭メタ情報に **アーティファクトURL** を明記する(パブリッシュ後に追記)

## アーティファクト(HTML)の記載規則

CLAUDE.md「レポート規約 → 読者別の表現使い分け」を厳守。アーティファクトの読者はユーザー本人のみ。

**書いてはいけないもの**: 内部ファイル名(`references/*` `plans/*` `programming.md §*`)、skill セクション番号や絶対規則番号、DB 表名(「ユーザー登録種目」等)、監修メモ(「概算・監修時要突合」等)、記号の内部説明(「★ = DB 未登録」等)。

**書き換えの例**:

| ❌ NG | ✅ OK |
|---|---|
| 週間ボリューム programming.md §1 の 10-20 セット帯内 | 部位別の週間セット数はトレーニング科学の推奨範囲内 |
| デロード判定: profile 体質特性 override により… | 疲労が溜まっているため、来週は少し軽めのセット数に調整するのがおすすめ |

**専門用語**: BMR・TDEE・PFC・RPE・Epley・デロード・1RM・レップ・セット・コンパウンド/アイソレーション等は初出時に日本語補足を必ず付ける。

**初心者・中級者向け表現ポリシー**(CLAUDE.md 参照):

- 専門用語1つに対して補足1つを徹底する
- 理由は"体で何が起きるか"で書く
- 「〜すべし」ではなく「〜すると良い」の口調(コーチが横で声をかけるトーン)
- 数字は日常語で言い直す
- できなかった時の代替案を1つ添える

## 評価セクション(§4 良かった / §5 惜しかった / §6 改善点・次アクション)の書き方

アーティファクト側では以下 3 要素を毎項目に含める(md 側は数値根拠のみで可):

1. **数値根拠**: 達成率・週次推移・前週比・基準値との差を具体的に
2. **なぜそうなったか**: 推定原因を1文(週次視点: セッション頻度・食事の日別偏り・睡眠時間の週内変動 等)
3. **来週どう動くか**: 動詞で始まる具体行動(「来週は月曜と木曜にたんぱく質を +30g 追加します」「金曜のトレーニング量を 8 割に落として金〜土でデロードします」)

抽象論(「デロード判断: 必要」等の単語句)は禁止。行動を動詞で書く。
