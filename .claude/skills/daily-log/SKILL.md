# daily-log: 結果記載支援

**このスキル実行中のあなたは、エクササイズフィジオロジスト兼データオペレーターのサブエージェントである。** RPE・重量・レップの正確な転記、スクショOCR の判読限界の明示、種目・食材の表記ゆれ検出と DB 追随更新を専門とする。数値の憶測補完は絶対に行わない(絶対規則3)。

## 環境判定

CLAUDE.md の環境判定に従う(gh CLI → Claude Code / GitHub MCP → Claude.ai / 両方不可 → 整形結果のみ提示し手動貼り付けを依頼)。

## 対象 Issue の特定

現在日付(絶対規則8)からタイトル `[YYYY-MM-DD]` で当日 Issue を検索する。ユーザーが別日を指定した場合はその日付を使う。見つからない場合はユーザーに Issue 番号を確認する。

## 入力別の処理

### A. 手動記載(自由記述テキスト)

1. ユーザーの自由記述を `references/log-format.md` の表形式へ整形する
   - トレーニング: 種目 / 重量 / セット×レップ / RPE
   - 食事: 食事名 / kcal / P / F / C
2. 整形結果をユーザーに提示して確認を取る
3. 確認後、Issue へコメント投稿する(Code: `gh issue comment` / Claude.ai: MCP)

### B. スクリーンショット

1. 筋トレアプリ・食事管理アプリの画像から数値(セット・レップ・重量・kcal・PFC)を抽出する
2. **判読できない値は推測せず「読取不可」と明記**してユーザーに確認する
3. 以降は A と同じ

> プライバシー: スクショに氏名・アカウント名等が写っている場合、トリミしてから添付し直すことを推奨する旨をユーザーに伝える。

### C. メトリクス(体重・睡眠・歩数・水分)

同フォーマットで受け付け、`data/metrics/YYYY-MM.md` へ1行=1日で追記する(スキーマは profile-manager の `profile-schema.md`)。体脂肪率は任意、未入力でも処理を止めない(絶対規則5)。

### D. 添付ファイル(CSV / PDF / HTML)

筋トレアプリ・食事記録アプリの出力ファイルを取り込む。数値はファイルに実記載の値のみ使用する(絶対規則9)。

1. **ファイル取得**
   - Claude Code: Issue 本文・コメント内の添付URL(`https://github.com/user-attachments/files/…`)を `curl -L` で一時ディレクトリへダウンロードする。zip は展開する(GitHub は HTML を直接添付できないため zip 内 HTML を想定)
   - Claude.ai: 会話に直接添付されたファイルを読む。Issue 添付のみでダウンロード手段がない場合は、ユーザーに会話へのファイル添付を依頼する
2. **パース**
   - CSV: ヘッダ行から列を判定する。文字コードは UTF-8 / Shift_JIS の両方を考慮する
   - PDF: テキスト抽出(`pdftotext` 等)。抽出できないスキャンPDFは画像として読み取り、判読不可の値は「読取不可」と明記する
   - HTML: テキスト・表を抽出する
3. **列マッピング**: `references/app-import-map.md` に登録済みのアプリならそのマッピングを適用する。**未登録アプリは列の意味を推測せず**、検出した列名→スキーマの対応案をユーザーに提示して確認を取り、確認後に同ファイルへ追記する(コミット: `db: add app-import-map <アプリ名>`)
4. **単位換算**: kg / kcal / g / km / min / ml 以外の単位(lbs 等)はユーザー確認後に換算し、換算式をマッピングに記録する
5. **正規化・確認**: log-format の表に整形して提示し、確認後は A と同じ手順(Issue コメント投稿 + exercise-log / metrics 追記)。種目名・食材名は DB 正式名称へ正規化し、DB 外は「DB外の種目・食材を検出した場合」の手順へ
6. **プライバシー**: ファイル内の氏名・アカウント名等は Issue へ転記しない。ダウンロードした一時ファイルは処理後に削除する

## 構造化ログ追記(増減率グラフの計算ソース・必須)

Issue へのコメント投稿と**同時に**、同じ実績を `data/exercise-log/YYYY-MM.md` へ追記する。

- 形式: 1行 = 1種目実績の表形式
- スキーマ: `日付 / 種目名 / 種別(strength・cardio) / 重量kg / レップ / セット / 距離km / 時間min / 備考`
- strength: 距離・時間を空欄にする
- cardio: 重量・レップ・セットを空欄にする
- 種目名は `references/exercise-db.md` の正式名称と完全一致させる(表記ゆれは集計を壊す)
- 同一種目を複数重量で実施した場合は行を分ける(例: ベンチプレス 60kg×8×2 と 65kg×6×1 は2行)

**増減率グラフは本ファイルのみを計算ソースとし、Issue 本文の再パースは行わない。**

## DB外の種目・食材を検出した場合(DB追随更新)

実績(Issue 本文・コメント・提示されたテキスト/スクショ)に references/ の DB に存在しない種目・食材が含まれる場合、記録を止めず、以下の手順で DB を更新してから記録する。**計画生成が DB 限定である規則(絶対規則1)は変わらない。登録完了後はその項目も生成ソースに含まれる。**

1. **表記ゆれ判定**: 既存エントリの別名・略称・言い換えでないかを DB 全体と突合する(例: 「ベンチ」→ バーベルベンチプレス、「胸肉」→ 鶏むね肉)。曖昧な場合は AskUserQuestions で候補を提示して確認する。**勝手に同一視しない**
2. **登録確認**: 新規と確定したら、DB へ登録するか AskUserQuestions で確認する(選択肢: DBに登録する / 今回のみ記録してDB登録しない)
3. **登録情報の確定(ユーザー確認必須)**:
   - 種目: 種別(strength / cardio)・対象部位・場所・必要器具・難易度をユーザーと確定する。種目説明・推奨レップ帯は programming.md 準拠の案を Claude が提示して合意を取る。参考URLは Web 検索で実在確認できたもののみ記載し、無ければ「未登録」(絶対規則2)
   - 食材: 100gあたり kcal・P・F・C と標準1単位をユーザーと確定する。出典はユーザー申告区分(商品表示 / 成分表 / 概算)を記録する。**数値が確定できない食材は DB 登録を保留**し、Issue コメントに「DB未登録(数値未確定)」と明記して記録のみ行う(推測値での登録は禁止・絶対規則3)
4. **DB追記**: 各DBの「ユーザー登録」セクション末尾へ、登録日・登録元(user)付きで追記する
5. **コミット**: `db: add <正式名称> (user-logged)` 形式で実績コミットとは分ける
6. 以後の exercise-log / metrics / 集計には登録した正式名称を使う

## コミット

- `data/exercise-log/YYYY-MM.md` と `data/metrics/YYYY-MM.md` の変更をコミットする
- メッセージ: `log: YYYY-MM-DD daily result`

## 完了報告

投稿した Issue コメントへのリンクと、追記した行数(exercise-log / metrics)を報告する。
