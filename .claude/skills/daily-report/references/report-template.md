## md の記載規則

- Mermaid ブロックは必須(文章のみ不可)。`xychart-beta` が GitHub でレンダリングされない場合に備え、同データの表を必ず併記する
- 増減率の計算ソースは `data/exercise-log/` のみ。定義は programming.md §5
- 評価(§4〜6)は数値根拠なしの記載を禁止
- 摂取カロリー・たんぱく質は許容幅(摂取カロリー −200 kcal / たんぱく質 −10 g)内なら達成扱いとし、**緑色(達成)で表示する**(黄色警告・「不足」表現は使わない)。惜しかった点・警告に挙げない。許容幅は不足方向のみ。達成率の数値は実測どおり表示する
- §9 総評(トレーナーからのまとめ)を必ず記載する。良かった点・惜しかった点を踏まえた要点の要約 + 明日の最優先1点を、数値根拠付きで書く
- 体調チェック(§2)・自分自身への言葉(§8)はユーザー記入内容をそのまま反映する。自分自身への言葉は原文引用とし、要約・改変・添削をしない
- 「身体の痛み」にチェックがある場合は傷病検知として扱い、専門家(医師・管理栄養士・トレーナー)への相談を促した旨を明記する(絶対規則6。医療・診断は行わない)
- md 側は skill セクション番号・DB 表名・出典ラベルの記載 OK(トレーサビリティ確保)
- md の冒頭メタ情報に **アーティファクトURL** を明記する(パブリッシュ後に追記)

## アーティファクト(HTML)の記載規則

CLAUDE.md「レポート規約 → 読者別の表現使い分け」を厳守。アーティファクトの読者はユーザー本人のみ(スマホ・PCで閲覧)。

**書いてはいけないもの**:
- 内部ファイル名(`references/nutrition-db.md`、`plans/2026-07/plan.md §5.1` 等)
- skill セクション番号や絶対規則番号(`programming.md §4.3`、`絶対規則3`、`§4.4 override 準拠` 等)
- DB 表名(「ユーザー登録種目」「ユーザー登録食材」等の内部運用語)
- 監修メモ(「概算・監修時要突合」「出典『概算』」等)
- 記号の内部説明(「★ = DB 未登録」等)

**書き換えの例**:

| ❌ NG | ✅ OK |
|---|---|
| BMR 1,889 に接近 · programming.md §4.3 下限規則注意 | 摂取カロリーが基礎代謝(BMR)1,889kcal ぎりぎりまで下がっています。これ以上減ると筋肉が落ちやすくなります |
| profile 体質特性 override 順守 | あなたの体質(脂質分解が苦手)に合わせた制限の範囲内 |
| 計画 4×6-10 · profile 必須種目 | 計画: 4 セット × 6〜10 回(必ず入れる種目) |
| ★ = DB 未登録 → references/nutrition-db.md ユーザー登録食材へ追加 | (アーティファクトには書かない。データ管理は md 側のみ) |

**専門用語**: BMR・TDEE・PFC・RPE・Epley・デロード・1RM・レップ・セット・コンパウンド/アイソレーション等は初出時に日本語補足(例: 「基礎代謝(BMR)」「主要栄養素の内訳(PFC=たんぱく質・脂質・炭水化物)」「主観的運動強度(RPE・1〜10で今日どれだけキツかったか)」「推定1RM(1回で挙げられる最大重量の推定値)」「多関節種目(コンパウンド)」)。

**初心者・中級者向け表現ポリシー**(CLAUDE.md 参照):

- 専門用語1つに対して補足1つを徹底する
- 理由は"体で何が起きるか"で書く(「メカニカルテンションが〜」ではなく「筋肉に強い張力をかけ続けることで太くなる」等)
- 「〜すべし」ではなく「〜すると良い」の口調(コーチが横で声をかけるトーン)
- 数字は日常語で言い直す(「体脂肪率 −0.8%/月」だけでなく「1か月で体重に対して 0.7kg 分の脂肪が減る計算」等)
- できなかった時の代替案を1つ添える(「懸垂が回数届かない日はアシストマシンで補助を強めに」等)

**体調チェック・自分自身への言葉のアーティファクト反映**:
- 体調チェックは「03 体の状態」の体調チェックカードへ反映する(問題ありは注意色、問題なしは控えめ表示。チェック記号やフォーム項目IDの内部説明は書かない)
- 自分自身への言葉は「05 今日の振り返り」冒頭の引用カードへユーザーの原文のまま掲載する(改変・要約禁止)。空欄時はカードに「今日は記入なし」と表示する
- 総評は「05 今日の振り返り」末尾の「■ 総評 — トレーナーから」カードへ、トレーナーがユーザーへ語りかけるトーンで掲載する。数値は一般語で表現し(内部用語禁止)、明日の最優先1点を含める
- プレースホルダと具体的な埋め方は `artifact-fill-guide.md` を参照する

## 評価セクション(§4 良かった / §5 惜しかった / §6 改善点・次アクション)の書き方

**アーティファクト側では以下 3 要素を毎項目に含める**(md 側は数値根拠のみで可):

1. **数値根拠**: 達成率・実測値・前日比・基準値との差を具体的に(「70kg × 6 回 = 推定1RM 84kg」「摂取 1,891 kcal は計画 2,770 に対し 879 kcal 不足」)
2. **なぜそうなったか**: 推定原因を1文(「1セット目直後で筋肉の回復が間に合わなかった可能性」「食事のタイミングが偏り、夜だけで詰め込む形になった」)
3. **明日どう動くか**: 動詞で始まる具体行動(「明日は次セットの休憩を 3 分に延ばします」「朝食のご飯を +100g、トレ前にオートミール +30g を追加します」)

抽象論(「疲労管理を強化」「食事バランス改善」等の名詞句)は禁止。行動を動詞で書く。

**総評(§9)の書き方**: 今日の要点を2〜4文でまとめ、良かった点は認め、要改善点は明日の具体行動に翻訳する。フィットネストレーナーがユーザーに語りかける口調で、数値根拠を1つ以上、明日の最優先アクションを1点必ず含める。専門用語は初出時に日本語補足を付ける。
```
